All routes live under `src/routes/`. Each file exports an Express `Router` that is mounted in `src/routes/index.ts`. Every route prefix inherits the `/api` prefix applied in `src/app.ts`.

## Barrel router: `src/routes/index.ts`

`src/routes/index.ts` acts as the single mount point for all sub-routers. Adding a route to the API means creating a new file and registering it here:

```typescript src/routes/index.ts
import { Router, type IRouter } from "express";
import healthRouter from "./health";

const router: IRouter = Router();

router.use(healthRouter);

export default router;
```

## Health check route

The only built-in endpoint is `GET /api/healthz`. It returns a JSON object confirming the server is running.

```typescript src/routes/health.ts
import { Router, type IRouter } from "express";
import { HealthCheckResponse } from "@workspace/api-zod";

const router: IRouter = Router();

router.get("/healthz", (_req, res) => {
  const data = HealthCheckResponse.parse({ status: "ok" });
  res.json(data);
});

export default router;
```

### Endpoint details

| Field | Value |
|---|---|
| Method | `GET` |
| Path | `/api/healthz` |
| Response | `{ "status": "ok" }` |
| Status code | `200` |

## Zod validation pattern

Route handlers use schemas from `@workspace/api-zod` to validate and shape response data. The `HealthCheckResponse` schema is generated from the OpenAPI spec by Orval:

```typescript lib/api-zod/src/generated/api.ts
// Generated by orval — do not edit manually.
import * as zod from "zod";

export const HealthCheckResponse = zod.object({
  status: zod.string(),
});
```

Calling `.parse()` on a schema serves two purposes:

1. **Runtime validation** — throws a `ZodError` if the data does not match the schema, catching accidental shape mismatches before they reach the client.
2. **Strip unknown fields** — Zod's default `parse` removes extra properties, preventing accidental data leakage.

<Note>
The generated schemas in `@workspace/api-zod` are the single source of truth. Do not define inline Zod schemas in route handlers — import from `@workspace/api-zod` instead.
</Note>

## Adding a new route

<Steps>
  <Step title="Create the route file">
    Create a new file in `src/routes/`. Export a typed `IRouter`:

    ```typescript src/routes/lessons.ts
    import { Router, type IRouter } from "express";
    import { LessonListResponse } from "@workspace/api-zod";

    const router: IRouter = Router();

    router.get("/lessons", async (_req, res) => {
      // fetch data from the database
      const raw = [{ id: 1, title: "Introduction" }];
      const data = LessonListResponse.parse(raw);
      res.json(data);
    });

    export default router;
    ```
  </Step>

  <Step title="Register the router in the barrel">
    Import and mount the new router in `src/routes/index.ts`:

    ```typescript src/routes/index.ts
    import { Router, type IRouter } from "express";
    import healthRouter from "./health";
    import lessonsRouter from "./lessons"; // add this

    const router: IRouter = Router();

    router.use(healthRouter);
    router.use(lessonsRouter); // add this

    export default router;
    ```

    The new endpoint is now reachable at `GET /api/lessons`.
  </Step>

  <Step title="Add the endpoint to the OpenAPI spec">
    Update `lib/api-spec/openapi.yaml` to document the new endpoint. Then run codegen to regenerate `@workspace/api-zod` and `@workspace/api-client-react`:

    ```bash
    pnpm --filter @workspace/api-spec run generate
    ```

    This keeps the server, validators, and client hooks in sync.
  </Step>
</Steps>

<Tip>
Express 5 handles `async` route handlers natively — thrown errors and rejected promises are forwarded to error-handling middleware automatically. You do not need to wrap handlers in a `try/catch` and call `next(err)` manually.
</Tip>
