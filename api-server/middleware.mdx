Middleware is registered in `src/app.ts` before routes are mounted. The server currently configures four global middleware layers.

## Configured middleware

```typescript src/app.ts
app.use(cors());
app.use(express.json());
app.use(express.urlencoded({ extended: true }));
```

| Middleware | Source | Purpose |
|---|---|---|
| `cors()` | `cors` package | Sets CORS response headers |
| `express.json()` | Express built-in | Parses `application/json` request bodies |
| `express.urlencoded()` | Express built-in | Parses `application/x-www-form-urlencoded` request bodies |
| `cookie-parser` | `cookie-parser` package | Parses `Cookie` header into `req.cookies` (installed, available to use) |

### CORS

CORS is enabled with default settings by passing no options to `cors()`:

```typescript
app.use(cors());
```

With default settings, `cors()` reflects any `Origin` header back in the `Access-Control-Allow-Origin` response header, which allows requests from all origins. For production deployments that need to restrict access, pass an options object:

```typescript
app.use(
  cors({
    origin: ["https://your-frontend.example.com"],
    credentials: true,
  }),
);
```

<Warning>
The default `cors()` configuration allows all origins. Restrict the `origin` option in production if your API should not be publicly accessible from any domain.
</Warning>

### JSON body parser

`express.json()` populates `req.body` for requests with a `Content-Type: application/json` header. It is a thin wrapper around the `body-parser` package that ships with Express.

### URL-encoded body parser

`express.urlencoded({ extended: true })` parses form submissions encoded as `application/x-www-form-urlencoded`. The `extended: true` option uses the `qs` library, which supports nested objects.

### cookie-parser

`cookie-parser` is listed as a dependency and is available to import, but is not yet mounted globally. Add it to `app.ts` when you need to read cookies:

```typescript src/app.ts
import cookieParser from "cookie-parser";

app.use(cookieParser());
```

## The `src/middlewares/` directory

The `src/middlewares/` directory is a placeholder for custom middleware modules. It is currently empty (contains only a `.gitkeep`). Place project-specific middleware here to keep `app.ts` clean.

## Adding custom middleware

<Steps>
  <Step title="Create a middleware file">
    Create a new file in `src/middlewares/`. A middleware function has the signature `(req, res, next) => void`:

    ```typescript src/middlewares/request-logger.ts
    import { Request, Response, NextFunction } from "express";

    export function requestLogger(
      req: Request,
      _res: Response,
      next: NextFunction,
    ): void {
      console.log(`${req.method} ${req.path}`);
      next();
    }
    ```
  </Step>

  <Step title="Mount the middleware in app.ts">
    Import the middleware and call `app.use()` before the route mounts:

    ```typescript src/app.ts
    import express, { type Express } from "express";
    import cors from "cors";
    import router from "./routes";
    import { requestLogger } from "./middlewares/request-logger";

    const app: Express = express();

    app.use(cors());
    app.use(express.json());
    app.use(express.urlencoded({ extended: true }));
    app.use(requestLogger); // add before routes

    app.use("/api", router);

    export default app;
    ```
  </Step>
</Steps>

## Express 5 and async errors

Express 5 changes how async errors are handled compared to Express 4.

In **Express 4**, unhandled promise rejections inside route handlers were silent — the request would hang unless you caught the error and called `next(err)` manually:

```typescript
// Express 4 — required explicit error forwarding
router.get("/example", async (req, res, next) => {
  try {
    const data = await fetchData();
    res.json(data);
  } catch (err) {
    next(err); // had to do this manually
  }
});
```

In **Express 5**, rejected promises and thrown errors inside `async` route handlers and middleware are automatically forwarded to error-handling middleware — no `try/catch` required:

```typescript
// Express 5 — async errors forwarded automatically
router.get("/example", async (_req, res) => {
  const data = await fetchData(); // rejection goes to error handler
  res.json(data);
});
```

To add a global error handler, register a four-argument middleware after all routes in `app.ts`:

```typescript src/app.ts
import { ErrorRequestHandler } from "express";

const errorHandler: ErrorRequestHandler = (err, _req, res, _next) => {
  console.error(err);
  res.status(500).json({ error: "Internal server error" });
};

// mount after all routes
app.use(errorHandler);
```
