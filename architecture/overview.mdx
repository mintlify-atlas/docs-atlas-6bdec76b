EduLearn Hub is a **pnpm workspace monorepo** running on Node.js 24. The backend is an Express 5 API validated end-to-end with Zod schemas that are generated automatically from a single OpenAPI specification. The frontend apps consume the same spec via generated React Query hooks — so the OpenAPI file is the single source of truth for every interface boundary in the stack.

## Request data flow

Every API call follows a straight path through the stack:

```text
Browser / React app
        │
        │  HTTP (fetch via customFetch)
        ▼
  Express 5 router  (/api/*)
        │
        │  Route handler  (src/routes/health.ts, …)
        ▼
  Zod schema validation  (@workspace/api-zod)
        │
        │  Validated, typed request data
        ▼
  Drizzle ORM  (@workspace/db)
        │
        │  SQL
        ▼
  PostgreSQL
        │
        │  Result rows
        ▼
  Drizzle result → Zod response schema
        │
        │  JSON
        ▼
  Express response  →  React Query cache
```

<Note>
Both the request _and_ response shapes are validated against Zod schemas derived from the same `openapi.yaml`. A schema mismatch fails at the TypeScript layer during codegen, not at runtime.
</Note>

## OpenAPI-first development cycle

The entire type system flows from one file: `lib/api-spec/openapi.yaml`. Editing that file and running codegen regenerates everything downstream automatically.

<Steps>
  <Step title="Edit the spec">
    Open `lib/api-spec/openapi.yaml` and add or modify paths, operations, and component schemas. The spec follows OpenAPI 3.1.

    ```yaml
    paths:
      /healthz:
        get:
          operationId: healthCheck
          tags: [health]
          summary: Health check
          responses:
            "200":
              content:
                application/json:
                  schema:
                    $ref: "#/components/schemas/HealthStatus"
    components:
      schemas:
        HealthStatus:
          type: object
          properties:
            status:
              type: string
          required:
            - status
    ```
  </Step>
  <Step title="Run codegen">
    Orval reads `openapi.yaml` via `orval.config.ts` and writes into two sibling packages:

    ```bash
    pnpm --filter @workspace/api-spec run codegen
    ```
  </Step>
  <Step title="Get typed Zod validators">
    `@workspace/api-zod` receives generated Zod schemas (e.g. `HealthCheckResponse`). The API server imports these for response validation.

    ```typescript
    // lib/api-zod/src/generated/api.ts — generated, do not edit
    export const HealthCheckResponse = zod.object({
      status: zod.string(),
    });
    ```
  </Step>
  <Step title="Get typed React Query hooks">
    `@workspace/api-client-react` receives generated hooks and a fetch client. Frontend apps import `useHealthCheck` or the plain `healthCheck` fetcher.

    ```typescript
    // lib/api-client-react/src/generated/api.ts — generated, do not edit
    export function useHealthCheck(options?) {
      const queryOptions = getHealthCheckQueryOptions(options);
      return useQuery(queryOptions);
    }
    ```
  </Step>
  <Step title="Typecheck the whole graph">
    Run the root typecheck to verify every package compiles with the new types:

    ```bash
    pnpm run typecheck
    ```
  </Step>
</Steps>

<Tip>
Never edit files inside `src/generated/` directories. They are overwritten on every `codegen` run. All changes start in `openapi.yaml`.
</Tip>

## How frontend apps consume the API

Frontend packages such as `@workspace/pi-day` declare a workspace dependency on `@workspace/api-client-react`:

```json
{
  "dependencies": {
    "@workspace/api-client-react": "workspace:*",
    "@tanstack/react-query": "catalog:"
  }
}
```

A component then imports the generated hook directly:

```typescript
import { useHealthCheck } from '@workspace/api-client-react';

function StatusBadge() {
  const { data, isLoading } = useHealthCheck();
  if (isLoading) return <span>Checking…</span>;
  return <span>{data?.status}</span>;
}
```

The hook calls the backend at `/api/healthz` using `customFetch`, which handles JSON content-type negotiation, BOM stripping, and throws typed `ApiError` instances on non-2xx responses.

## Security boundaries and environment variables

<Warning>
The API server will refuse to start if `DATABASE_URL` is not set. Never commit this value; it must be supplied by the deployment environment.
</Warning>

| Variable | Required by | Purpose |
|---|---|---|
| `DATABASE_URL` | `@workspace/db` (Drizzle config) | PostgreSQL connection string |
| `PORT` | `@workspace/api-server` (`src/index.ts`) | HTTP port the Express server binds to |

The Express app mounts CORS middleware (`cors` package) in `src/app.ts` before all routes. All routes are prefixed with `/api` — no route is accessible at the root path.

<Info>
In development, `DATABASE_URL` is automatically provisioned by the Replit environment. When running locally you must export it yourself before starting the dev server.
</Info>
