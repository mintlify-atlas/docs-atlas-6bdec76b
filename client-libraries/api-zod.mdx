`@workspace/api-zod` provides auto-generated [Zod](https://zod.dev) schemas and TypeScript types derived from the OpenAPI spec. The `api-server` package imports these schemas to validate and shape response data before sending it to clients.

<Note>
This package is auto-generated. Do not edit files inside `src/generated/` directly. To add or change a schema, update `lib/api-spec/openapi.yaml` and run codegen.
</Note>

| Field | Value |

| Zod version | v4 (via `zod` catalog entry) |

## Exports

`src/index.ts` re-exports everything from the two generated modules:

```typescript src/index.ts
export * from "./generated/api";    // Zod schemas
export * from "./generated/types"; // TypeScript interfaces
```

You only ever need to import from the package root:

```typescript
import { HealthCheckResponse } from "@workspace/api-zod";
import type { HealthStatus } from "@workspace/api-zod";
```

## Type inference

Derive the TypeScript type from the Zod schema using `z.infer`:

```typescript
import { z } from "zod";
import { HealthCheckResponse } from "@workspace/api-zod";

type HealthCheckResponseType = z.infer<typeof HealthCheckResponse>;
// { status: string }
```

This keeps the inferred type and the runtime schema permanently in sync — any change to the schema automatically updates the type.

## Using schemas in route handlers

Call `.parse()` on a schema in each route handler to validate the response payload before sending it. If the data does not conform to the schema, Zod throws a `ZodError` at runtime.

```typescript src/routes/health.ts
import { Router, type IRouter } from "express";
import { HealthCheckResponse } from "@workspace/api-zod";

const router: IRouter = Router();

router.get("/healthz", (_req, res) => {
  const data = HealthCheckResponse.parse({ status: "ok" });
  res.json(data);
});

export default router;
```

`.parse()` does two things:

1. **Runtime validation** — throws a `ZodError` if the shape is wrong, catching accidental mismatches before they reach the client.
2. **Strip unknown fields** — Zod's default parse removes extra properties, preventing unintended data from leaking in responses.

<Tip>
Use `.safeParse()` if you want to handle validation errors without throwing. It returns `{ success: true, data }` or `{ success: false, error }` instead of throwing on failure.
</Tip>

<Note>
The generated schemas in `@workspace/api-zod` are the single source of truth for response shapes. Do not define inline Zod schemas in route handlers — always import from `@workspace/api-zod`.
</Note>

## Updating generated schemas

<Steps>
  <Step title="Edit the OpenAPI spec">
    Modify `lib/api-spec/openapi.yaml` to add, change, or remove schemas.
  </Step>
  <Step title="Run codegen">
    ```bash
    pnpm --filter @workspace/api-spec run generate
    ```

    This overwrites all files in `src/generated/` in both `@workspace/api-zod` and `@workspace/api-client-react`.
  </Step>
  <Step title="Check TypeScript">
    ```bash
    pnpm run typecheck
    ```
  </Step>
</Steps>

See [Codegen](/development/codegen) for a full walkthrough of the generation pipeline.
