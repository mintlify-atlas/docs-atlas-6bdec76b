The `@workspace/mockup-sandbox` package is a lightweight Vite dev server used to preview individual React components in isolation. It auto-discovers every `.tsx` file in `src/components/mockups/` and makes each one accessible at a URL without any manual registration.

## How it works

<Steps>
  <Step title="Plugin scans the mockups directory">
    When Vite starts (or a file is added/removed), the `mockupPreviewPlugin` runs `fast-glob` over `src/components/mockups/**/*.tsx` and generates a module manifest at `src/.generated/mockup-components.ts`.

    ```typescript mockupPreviewPlugin.ts
    async function discoverComponents(): Promise<Array<DiscoveredComponent>> {
      const files = await glob(`${MOCKUPS_DIR}/**/*.tsx`, {
        cwd: root,
        ignore: ["**/_*/**", "**/_*.tsx"],
      });
      return files.map((f) => ({
        globKey: "./" + f.slice("src/".length),
        importPath: path.posix.relative("src/.generated", f),
      }));
    }
    ```

    Files and directories whose names start with `_` are excluded from discovery.
  </Step>
  <Step title="Generated manifest is imported by App.tsx">
    The manifest exports a `modules` map — an object keyed by component path, where each value is a dynamic-import factory:

    ```typescript src/.generated/mockup-components.ts
    // Auto-generated by mockupPreviewPlugin.ts — do not edit
    type ModuleMap = Record<string, () => Promise<Record<string, unknown>>>;
    export const modules: ModuleMap = {
      "./components/mockups/MyComponent.tsx": () => import("../components/mockups/MyComponent.tsx"),
    };
    ```
  </Step>
  <Step title="App routes the URL to the right component">
    `App.tsx` reads `window.location.pathname`. If the path matches `/preview/<name>`, it renders `PreviewRenderer` with that name. Otherwise it shows the `Gallery` landing page.

    ```tsx src/App.tsx
    function App() {
      const previewPath = getPreviewPath();
      if (previewPath) {
        return <PreviewRenderer componentPath={previewPath} modules={discoveredModules} />;
      }
      return <Gallery />;
    }
    ```
  </Step>
  <Step title="PreviewRenderer dynamically imports and mounts the component">
    `PreviewRenderer` looks up the path in the manifest, imports the module, and resolves the component using a priority order: `default` export → `Preview` named export → export matching the filename → last function export.

    ```tsx src/App.tsx
    function _resolveComponent(mod, name) {
      return (
        mod.default ||
        mod.Preview ||
        mod[name] ||
        Object.values(mod).filter((v) => typeof v === "function").at(-1)
      );
    }
    ```
  </Step>
</Steps>

## URL pattern

| URL | Result |
|---|---|
| `/` | Gallery landing page with usage instructions |
| `/preview/MyComponent` | Renders `src/components/mockups/MyComponent.tsx` |
| `/preview/subfolder/MyComponent` | Renders `src/components/mockups/subfolder/MyComponent.tsx` |

<Note>
The server requires `PORT` and `BASE_PATH` environment variables. It will throw at startup if either is missing.
</Note>

## Running the server

```bash
pnpm --filter @workspace/mockup-sandbox run dev
```

The dev server binds to `0.0.0.0` on the port given by `PORT`. Open the root URL to see the gallery, then navigate to `/preview/ComponentName` for any component.

```bash
# Type-check without emitting
pnpm --filter @workspace/mockup-sandbox run typecheck

# Build for production
pnpm --filter @workspace/mockup-sandbox run build
```

## Adding a new mockup component

<Steps>
  <Step title="Create the file">
    Add a new `.tsx` file under `src/components/mockups/`:

    ```tsx src/components/mockups/MyComponent.tsx
    export default function MyComponent() {
      return <div>Hello from MyComponent</div>;
    }
    ```

    The component must have at least one exported function. A default export is recommended — it will be resolved first.
  </Step>
  <Step title="The manifest updates automatically">
    The `mockupPreviewPlugin` watches the directory with `chokidar`. When you save the new file, it regenerates `src/.generated/mockup-components.ts` and Vite's HMR picks up the change without a full reload.
  </Step>
  <Step title="Open the preview">
    Navigate to `http://localhost:PORT/preview/MyComponent` to see the component rendered in isolation.
  </Step>
</Steps>

<Tip>
To exclude a component from auto-discovery (for example, a shared helper), prefix the filename or its parent directory with an underscore: `_MyHelper.tsx` or `_internal/SomeComponent.tsx`.
</Tip>

## Vite plugin details

`mockupPreviewPlugin.ts` is a Vite plugin with `enforce: "pre"`. It:

- Runs an initial `refresh()` scan on `buildStart`
- Starts a `chokidar` watcher on the mockups directory in `configureServer`
- Re-generates the manifest when files are added or removed
- Debounces concurrent refreshes: if a refresh is already in flight, the next call is queued and runs after the current one finishes
- Registers a Vite middleware that triggers an auto-rescan on `404` responses for paths containing `/components/mockups/` or `/.generated/mockup-components`

```typescript mockupPreviewPlugin.ts
export function mockupPreviewPlugin(): Plugin {
  return {
    name: "mockup-preview",
    enforce: "pre",
    async buildStart() {
      await refresh();
    },
    async configureServer(viteServer) {
      await refresh();
      // chokidar watches MOCKUPS_DIR for add/unlink events
      watcher = chokidar.watch(mockupsAbsDir, { ignoreInitial: true });
      watcher.on("add", (file) => { if (isMockupFile(file)) void onFileAddedOrRemoved(); });
      watcher.on("unlink", (file) => { if (isMockupFile(file)) void onFileAddedOrRemoved(); });
    },
  };
}
```

## Environment variables

| Variable | Required | Description |
|---|---|---|
| `PORT` | Yes | TCP port the Vite dev server binds to |
| `BASE_PATH` | Yes | Base URL path passed to Vite's `base` config option |

Both variables are validated at startup in `vite.config.ts`. If either is missing or `PORT` is not a valid positive integer, the process exits before Vite initializes.
