EduLearn Hub is a **TypeScript monorepo** that serves as the foundation for building interactive educational web applications. It brings together a production-ready backend, a type-safe database layer, auto-generated client libraries, and rich React frontends — all managed in a single pnpm workspace.

## Tech stack

| Layer | Technology |
|---|---|
| Package manager | pnpm workspaces |
| Language | TypeScript 5.9 |
| API framework | Express 5 |
| Database ORM | Drizzle ORM |
| Database | PostgreSQL |
| Validation | Zod v4 (`zod/v4`) |
| API codegen | Orval |
| Frontend | React 19 + Vite |
| UI components | Radix UI + Tailwind CSS v4 |
| State management | TanStack Query v5 |
| Bundler | esbuild |

## Monorepo structure

The repository is organized into three top-level directories: `artifacts/` for deployable applications, `lib/` for shared libraries, and `scripts/` for utility scripts.

```text
artifacts-monorepo/
├── artifacts/
│   ├── api-server/         # @workspace/api-server — Express 5 REST API
│   ├── mockup-sandbox/     # Component and UI mockup playground
│   └── pi-day/             # Interactive Pi Day frontend app
├── lib/
│   ├── api-spec/           # @workspace/api-spec — OpenAPI spec + Orval config
│   ├── api-client-react/   # @workspace/api-client-react — generated React Query hooks
│   ├── api-zod/            # @workspace/api-zod — generated Zod schemas
│   └── db/                 # @workspace/db — Drizzle ORM schema + DB connection
├── scripts/                # @workspace/scripts — utility scripts
├── pnpm-workspace.yaml
├── tsconfig.base.json      # Shared TS options (composite, bundler resolution, es2022)
├── tsconfig.json           # Root TS project references
└── package.json
```

## Key design decisions

### TypeScript composite projects

Every package extends `tsconfig.base.json`, which sets `composite: true`. The root `tsconfig.json` lists all packages as project references. This means:

- **Always typecheck from the root** — run `pnpm run typecheck`, which executes `tsc --build --emitDeclarationOnly`. This builds the full dependency graph so cross-package imports resolve correctly.
- **`emitDeclarationOnly`** — only `.d.ts` files are emitted during typecheck. Actual JS bundling is handled by esbuild, tsx, or Vite — not `tsc`.
- **Project references** — when package A depends on package B, A's `tsconfig.json` must list B in its `references` array. `tsc --build` uses this to determine build order and skip up-to-date packages.

<Warning>
Running `tsc` inside a single package will fail if its dependencies haven't been built yet. Always run typechecking from the workspace root.
</Warning>

### OpenAPI-first API development

The single source of truth for the API contract is `lib/api-spec/openapi.yaml`. From this file, Orval generates two outputs automatically:

- **`lib/api-zod/`** — Zod schemas used by the API server for runtime request and response validation.
- **`lib/api-client-react/`** — React Query hooks and a typed fetch client used by frontend applications.

This ensures the server, generated validators, and client hooks are always in sync with the OpenAPI spec.

### esbuild for production bundling

The API server uses esbuild to produce a single `dist/index.cjs` bundle. It bundles an allowlist of dependencies (express, cors, pg, drizzle-orm, zod, etc.) and externalizes the rest, resulting in a self-contained artifact ready for deployment.

## Root scripts

```bash
# Typecheck all packages using project references
pnpm run typecheck

# Typecheck then build all packages that define a build script
pnpm run build
```
